Abstract

In this report, I summarize the work done for my Google Summer of Code project with the Liquid Galaxy Project. The aim was to build the Liquid Galaxy Tour Management Server — a Node.js / Express.js backend that simplifies management, execution, and monitoring of KML (Keyhole Markup Language) tours on a Liquid Galaxy installation. The system acts as a centralized control server so users can upload, organize, edit, and launch virtual tours on the Liquid Galaxy cluster through a RESTful API and real-time communication.

Liquid Galaxy is a multi-display visualization platform powered by Google Earth, where synchronized computers render immersive panoramic views. Traditionally, interacting with the system requires manual file transfers and command execution. This project automates those tasks with a secure backend that abstracts the complexity of the underlying infrastructure.

Problem Statement

Managing KML tours in a Liquid Galaxy environment involved many manual steps:

  • Uploading KML files to the master machine
  • Organizing hundreds of tour files
  • Executing SSH commands manually
  • Monitoring execution status
  • Synchronizing commands across multiple display nodes
  • Handling user authentication

These manual operations increase configuration errors and reduce operational efficiency. The objective was a centralized backend capable of automating these operations through REST APIs and WebSocket communication.

Objectives

  • Build a RESTful API using Express.js.
  • Upload and manage KML / KMZ tour files.
  • Organize tours into folders with search, tags, and favorites.
  • Execute tours remotely on Liquid Galaxy via secure SSH.
  • Provide real-time status updates using Socket.IO.
  • Maintain user authentication (JWT, bcrypt, roles).
  • Store metadata and execution logs in MongoDB.
  • Expose dashboard statistics for operators.
  • Design an easily extensible backend architecture.

Project Work

  • 1. User Authentication

    Only authorized users can access the server. Implemented login / logout, JWT authentication, password hashing with bcrypt, role management, and Express middleware guards so tour upload and execution endpoints stay protected.

  • 2. KML Upload Module

    Users upload KML tour files through REST APIs (POST /api/tours/upload). Supported single KML and KMZ uploads, file validation, duplicate detection, metadata extraction, and file-size limits for .kml / .kmz formats via Multer.

  • 3. Tour Management Module

    After uploading, users organize tours: rename, delete, move folders, search, tag, and favorite. MongoDB stores tour name, description, author, upload date, category, tags, and file location for reliable cataloging across large tour libraries.

  • 4. REST API Layer

    Express.js exposes JSON APIs for the frontend dashboard, including: GET /api/tours, GET /api/tours/:id, POST /api/tours, PUT /api/tours/:id, DELETE /api/tours/:id, and POST /api/tours/run/:id. Responses follow a consistent success/message shape for easy client integration.

  • 5. SSH Command Engine

    The server connects securely to the Liquid Galaxy master machine using the ssh2 package. Workflow: Node server → SSH → master machine → Google Earth → display nodes. Functions include connect, authenticate, execute shell commands, upload KML, restart services, and trigger fly-to events — removing the need for manual SSH sessions.

  • 6. Real-Time Communication (Socket.IO)

    Socket.IO provides instant updates for upload progress, tour execution status, connected users, cluster health, and notifications. Events include tourUploaded, tourRunning, tourCompleted, tourFailed, and clusterConnected.

  • 7. Tour Execution Engine

    This module sends KML commands to Google Earth. Pipeline: receive API request → validate tour → generate KML command → SSH connection → transfer file → execute command → Google Earth opens tour → status returned to the client.

  • 8. Logging System

    Every activity is logged (user login, upload, delete, execution, errors, SSH commands) and stored in MongoDB with user, action, status, timestamp, and IP address for auditability.

  • 9. Search & Dashboard Statistics

    Search supports keyword, category, date, and author filters (e.g. GET /api/search?q=historical). Dashboard analytics expose total tours, active users, upload count, executed tours, failed executions, and storage usage.

  • 10. Security & Project Structure

    Security features: JWT auth, password hashing, HTTPS support, file-type validation, rate limiting, input validation (Joi), CORS, Helmet, and secure SSH authentication. Codebase organized into controllers, routes, middleware, models, services, sockets, uploads, config, utils, logs, and app.js / package.json.

Database Design

Conclusion

Working with the Liquid Galaxy Project was a challenging and rewarding experience. The Tour Management Server turns a previously manual KML workflow into an API-driven, authenticated, and observable backend suitable for museum and lab operators. Almost all planned objectives — REST APIs, upload/management, SSH execution, real-time status, MongoDB metadata, and logging — were implemented in an extensible Node.js architecture. I learned a great deal about Liquid Galaxy operations, secure remote execution, Socket.IO event design, and structuring production Express services.
